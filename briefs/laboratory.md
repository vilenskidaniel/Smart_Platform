# Бриф Продуктового Блока: Laboratory

## Назначение

`Laboratory` - это пользовательский инженерный контур платформы для модульной проверки, диагностики, первичного запуска и подбора рабочих параметров.

Внутренний контекст этапа и маршрута для совместимости пока может сохранять:

- старый псевдоним:
  - `service_test`
- рабочий маршрут:
  - `/service`

Но пользовательское имя для интерфейса теперь фиксируем как:

- `Laboratory`

Новые документы и файлы промптов должны использовать именно `Laboratory` как
каноническое имя блока. Старый псевдоним нужен только для слоя совместимости.

## Что это значит для UX

`Laboratory` не должен выглядеть как россыпь отдельных сервисных страниц.

Это одно общее рабочее пространство с общим каркасом и специализированными карточками внутри:

- первый уровень строится по каноническим инженерным категориям;
- второй уровень внутри категории строится по веткам срезов и рабочим карточкам;
- язык состояний, `Записи сессии`, `Конструктор` и `Сохранить выбор -> Settings` остаются общими для всего рабочего пространства;
- под ними меняется рабочая область;
- в рабочей области всегда должны быть:
  - состояние
  - элементы взаимодействия
  - видимая реакция системы и результат
  - подсказки по ожидаемому результату

Визуальное направление для `v1`:

- язык интерфейса ближе к приложению, а не к россыпи HTML-страниц;
- навигация без полной перезагрузки всего экрана;
- браузер на настольном экране - главный рабочий экран для глубокой инженерной работы;
- телефон остается честной точкой входа для статусов, выбора категории и базовых безопасных действий;
- штатная работа в полноэкранном режиме;
- разные режимы раскладки для обычного и полноэкранного режима без потери общего контекста;
- полная видимость даже неподдерживаемых модулей и правил.

## Обязательная логика доступности

- если модуль или подсистема недоступны, они не скрываются;
- они становятся серыми;
- поверх них должен быть серый кликабельный слой;
- при нажатии пользователь получает понятное объяснение:
  - почему модуль недоступен
  - что нужно сделать для разблокировки

## Что Уже Зафиксировано Как Базовый Набор

- единая точка входа:
  - `/service`
- `Laboratory` доступна и на `ESP32`, и на `Raspberry Pi`;
- внутри уже есть рабочее пространство с учетом владельца узла;
- первый углубленный срез уже существует для `strobe`;
- будущие срезы должны повторять ту же модель, а не разъезжаться по разным стилям.
- аппаратный источник истины по наличию и владению держим в `knowledge_base/resources/smart_platform_workshop_inventory.xlsx`.

## Первая обязательная структура вкладок

Первый уровень строится по крупным категориям, а не по владельцу узла:

- `Обзор`
- `Свет`
- `Приводы`
- `Вода`
- `Аудио`
- `Датчики`
- `Камера`
- `Дисплеи`
- `Экспериментальное`

Второй уровень внутри категории строится по группам железа и функций:

- `Строб`
- `ESP32 / Стендовый строб`
- `Сервоприводы`
- `Шаговый двигатель / приводы`
- `Распыление / вода`
- `Инженерный срез полива`
- `Влажность почвы + клапаны + перистальтика`
- `Пьезо / ультразвук`
- `Голос / динамик`
- `Лидар`
- `Датчик движения`
- `Температура и влажность воздуха`
- `Камера`
- `Контур зрения`
- `Сенсорный дисплей Raspberry Pi`
- `Профиль дальномера HC-SR04`
- `Прием нового модуля`

Важно:

- `Ultrasonic` здесь означает tweeter / emitter laboratory slice;
- профиль класса `HC-SR04` остается экспериментальной частью `Lidar`, а не подменяет `Ultrasonic`;
- `Servos` фиксируются вокруг рабочего turret-baseline `MG996R + PCA9685`;
- `Stepper Motor / Drives` - strictly laboratory-only ветка и не часть turret motion UX;
- `Lidar` должен уметь тестировать owner-side `TFmini Plus` и laboratory-профили класса `HC-SR04`;
- `Displays` держит owner-side `Raspberry Pi Touch Display` отдельно от `Camera` и остается видимо заблокированным в оболочке только на `ESP32`, пока `Raspberry Pi` недоступен;
- `Laboratory` должна уметь принимать и внеплановые / неизвестные модули без слома общей структуры.

Верхний статус-бар Laboratory должен показывать:

- какие платы сейчас участвуют в системе;
- какие узлы на стороне владельца доступны;
- какие модули активны, ухудшены или закрыты политикой.

## Роль Laboratory

`Laboratory` нужен не только для разовых тестовых нажатий.

Это место, где пользователь должен:

- проверять отдельные подсистемы;
- подбирать тайминги, частоты, уровни и сценарии;
- начинать и завершать явную инженерную сессию с оператором, целью, аппаратным профилем и при необходимости внешним модулем;
- явно видеть контекст питания для модулей, чувствительных к bench-режиму;
- видеть реакцию системы и результат прямо в рабочей карточке;
- фиксировать `pass`, `warn`, `fail`, `blocked`, `skipped` и заметки оператора в `Записях сессии`, а не напрямую в `Gallery > Reports`;
- затем сохранять рабочие параметры.

Внутри этого же рабочего пространства должен жить явный слой `Конструктор` для:

- приема нового или неизвестного модуля;
- сборки черновой карточки из типовых laboratory-блоков;
- подготовки кандидата к просмотру и переносу в `Settings`.

Дополнительное правило для mixed-shell режима:

- оболочка `Raspberry Pi` хранит канонический серверный контур laboratory-сессии и синхронизирует его через `/api/v1/laboratory/session*`;
- зеркальные оболочки, например `ESP32`, могут держать черновую сессию локально, но смысл для пользователя остается тем же: легкий локальный рабочий след без прямого моста в `Reports`;
- это нужно, чтобы внутренний инженерный контур оставался единым даже при временном участии сторонних модулей или передаче управления владельцу узла.

## Реальные Кодовые Опоры

- `raspberry_pi/web/service.html` - главный лабораторный экран на стороне `Raspberry Pi`.
- `raspberry_pi/server.py` - текущий входной сервер, который реально отдает `/service` и соседние laboratory-маршруты.
- `raspberry_pi/laboratory_session.py` - текущая модель контекста сессии с полями владельца узла, питания, режима просмотра и активного инструмента.
- `raspberry_pi/laboratory_readiness.py` - текущая логика preflight и порядка запуска аппаратных проверок.
- `firmware_esp32/data/service/index.html` и `firmware_esp32/src/web/WebShellServer.cpp` - зеркальный и запасной слой `ESP32`, который не должен откатываться к старому плоскому запускателю.

Следующая обязательная стадия:

- перейти от разовых значений к единому контракту профилей и пресетов платформы;
- не применять эти значения глобально автоматически;
- дать оператору явный контур просмотра и применения для повышения laboratory-профиля до системной настройки;
- применять сохраненные параметры в:
  - `automatic`
  - `manual FPV`
  - повторных laboratory-сценариях

## Базовый Контекст Питания

Для срезов, завязанных на регулируемый лабораторный блок питания, фиксируем явный ручной контекст:

- `Bench PSU`
- `LiFePO4 battery`

Это нужно, чтобы:

- не притворяться, что shell знает умные особенности зарядки, когда питание идет от батареи;
- честно блокировать или снижать объем калибровочного прохода, чувствительного к напряжению, в battery-context;
- оставлять видимую для оператора причину и путь возврата к полноценному bench-режиму.

## Базовый Аудиоконтур

Для будущего `audio`-среза фиксируем:

- все три аудиогруппы тестируются независимо;
- интерфейс при этом остается одной общей audio-вкладкой;
- текущая рабочая модель хранения:
  - один общий рабочий профиль
- Bluetooth-колонка должна стремиться автоматически переподключаться к активному узлу:
  - `Raspberry Pi` preferred
  - `ESP32` fallback requirement

## Следующие практические приоритеты

1. Довести `strobe` до laboratory-среза, готового к профилям.
2. Формализовать модель вкладок по железу и функциям для всего `Laboratory`.
3. Добавить экспериментальные профили для дальномеров класса `HC-SR04` и шаговых приводов.
4. Развести product UX и deep diagnostic UX окончательно.
5. После этого брать `audio` как следующий отдельный module-tab.
