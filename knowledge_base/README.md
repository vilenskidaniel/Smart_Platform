# Smart Platform Knowledge Base

Этот каталог становится новым активным слоем документации.

Его задача:

- держать компактный и читаемый человеческий канон;
- собирать смысл из legacy donor layer через перенос `вырезать из донора -> вставить в новый файл`;
- удерживать порядок чтения от общего к частному;
- не плодить параллельные каноны между новым слоем, prompt-слоем и кодом.

## Статус

- текущий статус: `skeleton active`
- legacy donor layer считается donor-only слоем
- прямое чтение legacy donor material допустимо только как работа с донором, а не как с активным каноном

## Главные Правила Переноса

1. Сначала наполняется новый файл.
2. Затем соответствующий старый donor-файл истончается или помечается как historical.
3. Если в доноре несколько конфликтующих формулировок, сравниваются варианты и выбирается лучший.
4. Любая сквозная договоренность сверяется с prompt-слоем, чтобы не появлялись два разных канона.
5. Кодовые примеры и UI-стандарты разрешены, если они помогают сделать правило однозначным.

## Стратегическая Поправка, Которую Нужно Удерживать В Новом Слое

Мы больше не считаем, что центральный управляющий узел `Irrigation` навсегда равен `ESP32`, а центральный управляющий узел `Turret` навсегда равен `Raspberry Pi`.

Новый канон:

- модуль описывается на уровне роли, состава, интерфейсов, ограничений и сценариев;
- конкретный controller node считается временным или текущим implementation-profile;
- controller node в этом смысле ближе к заменяемому компоненту платформы, чем к вечной архитектурной истине модуля.

## Требование К Prompt-Слою

Prompt-слой не должен хранить уникальную продуктовую истину, которой нет здесь.

Если правило уже стало устойчивым, оно должно быть записано в `knowledge_base/`, а prompt должен только:

- направлять чтение;
- удерживать дисциплину работы;
- не давать чату вернуться к donor-only слою как к живому канону.
