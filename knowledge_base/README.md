# Smart Platform Knowledge Base

Этот каталог является новым активным слоем документации и главной человеческой точкой входа в проект.

Его задача:

- держать компактный и читаемый человеческий канон;
- удерживать уже собранный канон после выноса смысла из legacy donor layer;
- удерживать порядок чтения от общего к частному;
- не плодить параллельные каноны между новым слоем, prompt-слоем и кодом.

## Статус

- текущий статус: `skeleton active`
- legacy donor layer уже удален из active repo
- historical migration state и остатки старой карты переноса держим в `17_open_questions_and_migration.md`, а не в отдельной donor-папке

## Главные Правила Переноса

1. Сначала наполняется новый файл.
2. Затем исторический источник теряет статус активной точки входа и остается только в migration-ledger или внешнем archival snapshot при необходимости.
3. Если в доноре несколько конфликтующих формулировок, сравниваются варианты и выбирается лучший.
4. Любая сквозная договоренность сверяется с prompt-слоем, чтобы не появлялись два разных канона.
5. Кодовые примеры и UI-стандарты разрешены, если они помогают сделать правило однозначным.

## Стратегическая Поправка, Которую Нужно Удерживать В Новом Слое

Мы больше не считаем, что центральный управляющий узел `Irrigation` навсегда равен `ESP32`, а центральный управляющий узел `Turret` навсегда равен `Raspberry Pi`.

Новый канон:

- модуль описывается на уровне роли, состава, интерфейсов, ограничений и сценариев;
- конкретный controller node считается временным или текущим implementation-profile;
- controller node в этом смысле ближе к заменяемому компоненту платформы, чем к вечной архитектурной истине модуля.

## Reading Order Для Человека

Читать так:

1. `01_project_scope_and_goals.md`
2. `02_system_terms_and_design_rules.md`
3. `03_platform_architecture_and_module_relationships.md`
4. `04_runtime_topology_controller_profiles_and_sync.md`
5. `05_shell_navigation_and_screen_map.md`
6. `06_shared_ui_contract.md`
7. `07_data_registry_storage_and_persistence.md`
8. `08_safety_acceptance_and_field_operations.md`
9. `09_repository_layout_and_code_map.md`
10. один целевой модульный или service-level файл из `10-16`
11. соответствующий prompt-файл из `chat_prompts/`
12. `17_open_questions_and_migration.md`, только если нужен historical migration status или карта прошлых переносов

Практическое правило:

- новый специализированный чат не должен собирать контекст из нескольких параллельных summary-слоев;
- после `01-09` выбирать один целевой модульный файл и один соответствующий prompt;
- если задача вылезает за один модуль, переходить в `chat_prompts/cross_module_prompt.md`, а не собирать новый reading order вручную.

## Структура Нового Активного Набора

- `01-09` — общий канон платформы
- `10-16` — модульные и прикладные deep-spec файлы
- `17` — открытые вопросы, миграция и donor-thinning

## Требование К Prompt-Слою

Prompt-слой не должен хранить уникальную продуктовую истину, которой нет здесь.

Если правило уже стало устойчивым, оно должно быть записано в `knowledge_base/`, а prompt должен только:

- направлять чтение;
- удерживать дисциплину работы;
- не давать чату вернуться к donor-only слою как к живому канону.
