# ESP32 Shell Bootstrap

Этот документ фиксирует результат текущего этапа:

- поднят первый `ESP32 web shell`;
- shell работает по `Wi-Fi`;
- shell читает данные из `System Core`;
- shell показывает турельные модули как `locked`, если `Raspberry Pi` недоступен.

## Что уже реализовано

### 1. Wi-Fi bootstrap

`ESP32` поднимает собственную точку доступа.

На текущем этапе это самый практичный вариант для батарейного и беспроводного старта:

- не нужен USB;
- не нужен внешний роутер;
- можно зайти на страницу прямо с телефона или ноутбука.

### 2. HTTP shell

Поднят простой web-сервер, который отдает:

- `/`
- `/api/v1/system`
- `/api/v1/modules`
- `/api/v1/diagnostics`

### 3. Shell UI

Страница уже показывает:

- локальный узел;
- peer-узел;
- активный режим;
- список модулей;
- `Irrigation`;
- `Turret`;
- `Service/Test`;
- `Diagnostics`.

### 4. Locked turret behavior

На `ESP32`:

- `turret_bridge` и `strobe` видны;
- они не скрываются;
- они помечаются как `locked`;
- причина блокировки: `owner_unavailable`.

Это соответствует базовой архитектуре платформы.

## Почему shell пока встроен в прошивку

Это сделано ради скорости старта.

Так проще:

- быстро собрать первый браузерный контур;
- проверить модель блокировок;
- не тормозить развитие `System Core`.

Но это не конечное решение.

## Следующий логичный шаг

После bootstrap shell стоит делать одно из двух:

1. `Module Registry + dynamic shell sections`
   Чтобы shell строился из реестра еще гибче и готовился к новым модулям.

2. `Strobe module integration`
   Чтобы первый реальный модуль новой платформы появился внутри service/test и общей модульной карты.

Мой приоритет:

- сначала довести shell до состояния "готов принимать модули";
- потом интегрировать `strobe` как первый настоящий модуль.
