# System Core Spec

Этот документ описывает второй этап master-plan:

- общий `System Core`;
- состояния узлов и модулей;
- глобальные режимы;
- fault и block причины;
- стартовое поведение `ESP32`-узла.

## Зачем нужен System Core

`System Core` нужен как единый слой, который понимают все части платформы:

- `ESP32`
- `Raspberry Pi`
- shell интерфейс
- синхронизация
- сервисные страницы

Без этого слоя каждая сторона начнет по-своему трактовать:

- что значит `online`;
- когда модуль `locked`;
- почему кнопка серого цвета;
- когда можно исполнять команду;
- кто сейчас владелец действия.

## Что входит в System Core

### 1. Local Node Health

Это состояние текущего узла:

- `node_id`
- `node_type`
- флаг локального узла
- сетевой статус
- готовность shell
- uptime

### 2. Peer Node Health

Это сокращенная информация о соседнем узле:

- доступен ли он;
- когда последний раз был heartbeat;
- известен ли его `node_id`;
- что делать с зависимыми от него модулями.

### 3. Active Mode

Текущий глобальный режим системы:

- `manual`
- `automatic`
- `service_test`
- `fault`
- `emergency`

### 4. Module Registry

Реестр всех модулей, известных shell интерфейсу.

Для каждого модуля хранятся:

- идентификатор;
- заголовок;
- владелец;
- профиль;
- группа UI;
- текущее состояние;
- причина блокировки;
- флаги возможностей;
- видимость в shell.

## Канонические состояния модулей

### `online`

Модуль доступен и может работать штатно.

### `degraded`

Модуль доступен частично. Он может показываться и выполнять часть действий, но не весь заявленный набор функций.

### `locked`

Модуль видим, но управление заблокировано.

Типичные причины:

- владелец недоступен;
- активен сервисный режим;
- действует межблочная защита;
- нужна дополнительная синхронизация.

### `fault`

Модуль находится в аварийном или ошибочном состоянии. Для продолжения требуется явное устранение причины.

### `service`

Модуль переведен в сервисный контур. Автоматическое использование должно быть запрещено или жестко ограничено.

### `offline`

Модуль в данный момент недоступен как функциональный объект.

## Канонические причины блокировки

На первом этапе фиксируем такой словарь `block_reason`:

- `none`
- `owner_unavailable`
- `peer_sync_pending`
- `safety_interlock`
- `module_fault`
- `service_session_active`
- `service_mode_required`
- `emergency_state`
- `module_offline`
- `unknown`

## Стартовое поведение ESP32

После первого запуска `ESP32` shell должен вести себя так:

- локальный узел считается `online`;
- `shell`, `settings`, `logs`, `diagnostics` и `irrigation` доступны локально;
- `turret_bridge` и `strobe` отображаются, но сразу помечаются как `locked`;
- причина блокировки: `owner_unavailable`;
- при появлении heartbeat от `Raspberry Pi` турельные модули могут перейти в `degraded` или `online` в зависимости от глубины синхронизации.

## Что именно реализуем на этапе 2

- перечисления состояний;
- типы для `node health` и `module descriptor`;
- локальный `module registry`;
- правила обновления турельных модулей при появлении или исчезновении peer-узла;
- минимальный отладочный вывод `system snapshot`.
