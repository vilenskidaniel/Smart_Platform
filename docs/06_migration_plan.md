# Migration Plan

## Цель миграции

Не "склеить" старые проекты в один большой файл, а аккуратно собрать новую платформу из лучших решений.

## Откуда что берем

### Из ESP32_COB_Strobe_Bench

- безопасное ядро модуля;
- state machine подход;
- event log;
- пресеты;
- диагностические снимки состояния;
- разделение UI и управления железом.

### Из IrrigationSystemESP32

- `Wi-Fi` и `SoftAP/STA` идеи;
- `LittleFS` и раздачу статики;
- структуру страниц и shell навигацию;
- идеи каталога растений и полива;
- подход к автономному браузерному управлению.

### Из IS with Server

Ничего не переносим напрямую.

## Что не переносим как есть

- слабую авторизацию;
- захардкоженные логины и пароли;
- токены и секреты в коде;
- монолитные `main.cpp`;
- недописанные заглушки без явной маркировки;
- страницы, которые напрямую предполагают доступ к аппаратуре без слоя модулей.

## Стратегия переноса

### Stage 0

Зафиксировать документы и контракты.

Включая:

- `docs/smart_platform_workshop_inventory.xlsx` как аппаратный источник истины;
- правило, что donor-репозитории используются только для сверки и выборочной миграции.
- legacy `ТЗ` `.docx` только как временный `knowledge donor`, а не как новый product spec.

### Stage 0.5

Выжать полезные `knowledge packets` из legacy `ТЗ` и разложить их по текущей документации `Smart Platform`.

Артефакт этого шага:

- `docs/42_docs_completeness_migration_plan.md`

Правило:

- сам legacy `.docx` не обновляем;
- переносим только еще полезные знания;
- после закрытия document-gaps legacy файл должен быть удален из активного проекта.

### Stage 1

Собрать общий каркас каталогов платформы.

### Stage 2

Собрать общее системное ядро.

### Stage 3

Поднять `ESP32` web shell и базовые страницы.

### Stage 4

Перенести модуль `strobe` в новом виде.

### Stage 5

Перенести и почистить `irrigation`.

### Stage 6

Собрать `Raspberry Pi` сторону и `turret_bridge`.

### Stage 7

Включить двустороннюю синхронизацию между узлами.

### Stage 8

Довести сервисный режим и модульное тестирование.

### Stage 9

После подтвержденной выборочной миграции очистить donor-layer из активного рабочего контекста.

## Что переписываем, а не копируем

- auth и доступ;
- сетевой контракт;
- общую карту страниц;
- логику владельцев модулей;
- логику синхронизации;
- объединенную модель статуса.
- старую структуру legacy `ТЗ` как целевой формат документации.
- локальные editor/env-конфиги donor-репозиториев тоже не переносим как часть новой платформы.

## Что можно адаптировать быстрее

- безопасную state machine для стробоскопа;
- event log и stop reasons;
- структуру веб-страниц и навигации;
- `SoftAP` и раздачу статики;
- часть сенсорной логики полива.
