# Stage 17 - Quick Audit And Backlog Snapshot

Этот документ нужен как короткий аудит текущего состояния `Smart Platform`.

## Что уже выглядит правильно

1. Есть последовательная архитектурная линия.
   - `ESP32` отвечает за fallback shell, локальные модули и safety-friendly режимы.
   - `Raspberry Pi` идет в роль владельца турели.
   - ownership модулей не плавает хаотично.

2. Фундамент не остался только на бумаге.
   - `System Core` уже живой.
   - `module registry` уже живой.
   - `web shell` уже живой.
   - `strobe_bench`, `irrigation` и `sync bootstrap` уже живые.

3. Каждая стадия подтверждается сборкой.
   - `pio run`
   - `pio run -t buildfs`
   - Python bootstrap можно отдельно проверять синтаксически.

4. Появилась нормальная база для параллельной работы из других чатов.
   - есть briefs;
   - есть contracts;
   - есть master-plan;
   - есть отдельные stage-документы.

## Что уже видно как долг

1. Документы местами отстают от кода.
   - это нужно регулярно подчищать, иначе другие чаты будут стартовать со старой картиной.

2. `ESP32`-сторона сейчас сильнее `Raspberry Pi`-стороны.
   - это нормально на данном этапе, но разрыв уже заметен.

3. Sync пока только bootstrap.
   - heartbeat есть;
   - push состояния есть;
   - полноценного merge логов, настроек и сценариев еще нет.

4. UI пока не вынесен в единый общий слой.
   - дизайн уже близкий;
   - но переиспользование общих ассетов между `ESP32` и `Raspberry Pi` еще не собрано.

5. Есть честные временные заглушки.
   - `irrigation` сейчас в `dry-run`;
   - `Raspberry Pi turret bridge` пока имитирует runtime, а не управляет реальной турелью.

## Есть ли сейчас "просадки и бред"

На текущем этапе явных просадок не видно.

Почему я так считаю:

- архитектура не расползается от шага к шагу;
- ownership модулей не ломается;
- shell и contracts двигаются в одну сторону;
- build-проверка продолжает проходить;
- временные места помечаются как `TODO(stage-...)`, а не маскируются под готовую систему.

Но есть риски, за которыми уже нужно следить:

- не плодить слишком много stage-документов без периодической консолидации;
- не дублировать UI слишком долго между `ESP32` и `Raspberry Pi`;
- не откладывать реальную двухузловую обкатку слишком надолго.

## Крупный список оставшихся работ

### Ближайшие

1. Довести `Raspberry Pi turret bridge` до уверенного bootstrap-уровня.
2. Научить `Raspberry Pi` честно публиковать runtime-состояние `turret_bridge` и `strobe`.
3. Сделать одинаковый shell UX на обоих узлах не только по виду, но и по структуре экранов.

### Средний горизонт

4. Реализовать двусторонний sync настроек.
5. Реализовать append-only sync логов.
6. Реализовать синхронизацию пользовательских сценариев и пресетов.
7. Перенести `Service/Test`-подход и на сторону `Raspberry Pi`.

### Дальше

8. Подключить реальный turret runtime.
9. Подключить реальные датчики и драйверы полива.
10. Вынести общий `web_ui` слой в реально переиспользуемые ассеты.
11. Подготовить полную двухузловую тестовую матрицу.
12. Провести живую обкатку на железе.
