# Sync And Ownership

## Уточнение от 2026-04-03: ownership по модулям, а не по всей системе

- Появление более мощного узла в сети не должно означать полный takeover всей платформы.
- Правильная модель для `v1`:
  - `ESP32` продолжает владеть `irrigation`, локальными датчиками, локальной SD-записью и своим сервисным контуром;
  - `Raspberry Pi` владеет turret-семейством модулей;
  - shell на любом узле показывает общую картину, но исполняет команды только через владельца конкретного модуля.
- Поэтому sync должен переносить не только state, но и сведения о том, где находится canonical owner page модуля.

## Уточнение от 2026-04-03: federated owner routing

- Для peer-owned модулей вводится owner-aware routing contract:
  - модуль знает своего владельца;
  - shell знает, доступен ли владелец;
  - shell знает canonical path модуля;
  - shell знает canonical owner URL, если peer уже в сети.
- На этом этапе допускается soft handoff:
  - показать, что owner доступен;
  - дать открыть owner page;
  - перенаправить сервисный или manual workflow к владельцу.
- Полный reverse-proxy можно добавить позже, но текущий UX уже должен строиться вокруг этой federated модели.

## Базовый принцип

Первая версия `Smart Platform` не использует полноценный выбор "самого производительного мозга".

Вместо этого система строится так:

- у каждого модуля есть понятный владелец;
- оба узла держат локальную копию состояния;
- состояния, настройки и журналы синхронизируются между узлами;
- при потере связи система продолжает жить в режиме controlled degradation.

## Что считается владельцем

Владелец модуля:

- принимает финальное решение об исполнении команды;
- хранит каноническое runtime-состояние модуля;
- отвечает за safety-проверки модуля;
- генерирует основные события и stop reasons.

## Канонические владельцы первой версии

В ownership-логике фиксируем не brand платы, а логическую роль узла.

- `io_node`
  - текущий тип платы может быть `ESP32`
  - `irrigation`
  - `strobe_bench`
  - часть `diagnostics`
  - fallback `platform_shell`

- `compute_node`
  - текущий тип платы может быть `Raspberry Pi`
  - `turret_bridge`
  - `strobe` в составе турели
  - тяжелая автоматика турели

- `shared`
  - `settings`
  - `logs`
  - `sync_core`
  - общие пользовательские параметры UI

Если в старом runtime или логах встречаются alias `esp32` и `rpi`, их можно
сохранять как compatibility-layer, но новые документы и UI vocabulary должны
мыслить ownership через `io_node`, `compute_node`, `shared`.

## Что именно синхронизируется

### Настройки

- язык
- тема
- базовые пользовательские предпочтения
- конфигурация страниц

### Логика системы

- пресеты
- пользовательские сценарии
- расписания
- чувствительности и поведенческие коэффициенты

### Runtime-данные

- heartbeat узлов
- состояние модулей
- active mode
- fault state
- сервисные блокировки

### Журналы

- локальные события
- аварии
- ручные действия пользователя
- срабатывания сценариев

## Правила разрешения конфликтов

### Безопасность

Безопасность не перетирается удаленно.

Если узел-владелец перевел модуль в `fault` или `locked`, второй узел может это только отобразить.

### Команды управления

Команду исполняет только узел-владелец модуля.

### Настройки

Для пользовательских настроек первой версии допустима схема:

- `last_write_wins`
- плюс метка времени
- плюс `source_node`

### Журналы

Журналы должны быть append-only.

Правильное поведение:

- каждый узел пишет локально;
- потом события объединяются при синхронизации;
- уже принятые записи не должны молча переписываться.

## Транспорт первой версии

Рекомендуемый старт:

- `HTTP JSON` для API;
- heartbeat и синхронизация короткими запросами по расписанию;
- браузер читает текущий узел напрямую;
- позже можно добавить `WebSocket`, если станет нужна более живая двусторонняя связь.

## Видимость страниц

Оба узла отдают одинаковый shell.

Разница в том, что:

- на `ESP32` турельные страницы видимы, но часто `locked`;
- на `Raspberry Pi` турель активна, а `ESP32`-страницы могут деградировать при потере базового модуля.
