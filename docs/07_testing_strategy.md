# Testing Strategy

## Общий принцип

Платформа проектируется так, чтобы тестировщики могли проверять модули отдельно, а не только весь комплекс разом.

Для этого вводится обязательный `Laboratory Mode`.

## Минимальные тестовые уровни

### 1. Тест отдельного модуля

Примеры:

- `strobe` как сервисный модуль;
- `irrigation` по зонам;
- `turret_bridge` без участия полива.

### 2. Тест владельца и блокировок

- на `ESP32` турель видна, но заблокирована;
- на `Raspberry Pi` полив уходит в деградацию при потере `ESP32`;
- команда не проходит, если модуль offline или fault.

### 3. Тест синхронизации

- смена темы и языка на одном узле;
- появление этого изменения на втором узле;
- объединение журналов;
- распространение статусов модулей.

### 4. Тест деградации

- отключен `Raspberry Pi`;
- отключена `ESP32`;
- плохой `Wi-Fi`;
- потеря heartbeat;
- конфликт сервисного и автоматического режимов.

### 5. Тест сценариев

- ручной сценарий;
- автоматический сценарий;
- сервисный сценарий;
- аварийная остановка.

## Особое внимание к стробоскопу

- безопасный старт;
- запуск сервисных пресетов;
- запрет запуска в неподходящем режиме;
- блокировка при fault;
- работа через владельца;
- корректное отображение на соседнем узле;
- остановка по `Abort`.

## Особое внимание к турели

- разблокировка только при наличии владельца `Raspberry Pi`;
- корректная блокировка на `ESP32`;
- управление стробом, распылением и звуком только через турельный слой;
- отказ при потере камеры или вычислительного модуля.

## Особое внимание к поливу

- автономная работа на `ESP32`;
- сохранность логики при отсутствии `Raspberry Pi`;
- сервисная проверка зон и датчиков;
- видимость ошибок и блокировок в общем shell.

## Turret Runtime Automated Checks

На стороне `Raspberry Pi` уже должны жить первые автоматические проверки, которые не зависят от реального железа:

- блокировка `strobe` в `automatic` без `target_locked`;
- аварийный `emergency` гасит активные актуаторы;
- service-сценарий `service_safe_idle` проходит успешно;
- turret events зеркалятся в общий `platform log`.

Эти проверки уже стоит держать как `unittest`, а не только как ручные smoke-проходы.

## Hardware Qualification Buckets

После появления реального списка компонентов тестирование нужно расширять не только по software-модулям,
но и по аппаратным пакетам.

### 1. Power Qualification

- проверка стартовых напряжений веток;
- проверка просадки при запуске actuator-классов;
- проверка, что логика не перезагружается при включении силовой нагрузки;
- проверка предсказуемого поведения при деградации питания.

### 2. Sensor Pack Qualification

- проверка чтения каждого датчика отдельно;
- проверка поведения при отсутствии датчика;
- проверка поведения при шумных или недостоверных данных;
- проверка, что `environment_pack` и `soil_sensing` корректно попадают в shell и журналы.

### 3. Water Path Qualification

- отдельная проверка `irrigation_water_path`;
- отдельная проверка `turret_water_path`;
- запрет смешивания команд двух водяных каналов;
- проверка аварийной остановки и восстановления.

### 4. Turret IO Qualification

- проверка `motion` в ручном и сервисном режиме;
- проверка `strobe` как turret-owned канала;
- проверка `audio`-подсистем;
- проверка блокировок при отсутствии владельца `Raspberry Pi`.

### 5. Camera And Vision Qualification

- проверка инициализации primary camera;
- проверка деградации при отсутствии камеры;
- проверка, что turret runtime не уходит в опасное состояние при отказе vision-stack;
- проверка sync-отражения camera-state на `ESP32`.
