# Переход К Модульным Чатам

Этот документ фиксирует лучший рабочий режим для следующего этапа проекта.

## Зачем Нужен Переходный План

Платформа уже доросла до состояния, где один длинный чат начинает хуже держать глубину и прозрачность.

Проблема не в том, что архитектура распадается, а в том, что:

- слишком много внутренних слоев начинают обсуждаться одновременно;
- product-level решения смешиваются с runtime-деталями;
- пользователю становится трудно контролировать, что именно сейчас прорабатывается.

Поэтому дальнейшая работа должна идти не "по всей платформе сразу", а по одному продуктовому блоку за чат.

## Напоминание По Контексту

На текущем этапе контекста хватает для:

- координации верхнего уровня;
- короткой уборки документации;
- сверки решений между модулями;
- итоговой интеграции после модульной проработки.

Но не стоит в одном чате подряд глубоко тянуть все большие направления сразу:

1. `System Shell v1`
2. `Irrigation v1`
3. `Turret v1`
4. `Gallery v1`
5. `Laboratory v1`
6. hardware integration
7. live integration
8. final cleanup

Такой режим снова приведет к разрастанию внутренней сложности и потере прозрачности.

## Рекомендуемый Порядок Следующих Чатов

1. `System Shell v1`
2. `Irrigation v1`
3. `Turret v1`
4. `Gallery v1`
5. `Laboratory v1`
6. возврат в координационный чат для синтеза
7. после синтеза: real hardware docs, live integration, итоговая уборка

## Важное Уточнение По Нумерации

Здесь легко перепутать два разных смысла:

- `Этап 2` master-plan — это `System Shell v1`;
- `второй пункт` модульной очереди после shell — это `Irrigation v1`.

На текущем срезе проекта `System Shell` еще не закрыт как полностью завершенный продуктовый этап,
но уже достаточно зрел как foundation для следующего отдельного чата.

Практический вывод:

- переход к `Irrigation v1` уже допустим;
- это не причина снова открывать большой общий shell-refactor,
  если shell не стал прямым блокером текущей задачи.

## Что Делать В Каждом Новом Чате

Каждый новый чат должен брать только один продуктовый блок:

- `System Shell`
- `Irrigation`
- `Turret`
- `Gallery`
- `Laboratory`

Если в ходе работы задача начинает затрагивать второй большой блок, это повод:

1. зафиксировать решение в docs;
2. завершить локальную часть;
3. открыть отдельный чат под следующий блок.

## Когда Возвращаться В Координационный Чат

Возврат нужен, если:

- завершен один product block;
- меняется ownership или federated handoff;
- нужно пересобрать общую картину;
- пора синхронизировать docs между несколькими модульными ветками;
- начинается живая интеграция `ESP32 + Raspberry Pi`.

Не нужно возвращаться туда слишком рано:

- если чат по `Irrigation v1` упирается только в локальную shell-правку,
  стоит исправить прямой блокер и продолжить модульную работу;
- если задача внезапно превращается в большой cross-module refactor,
  тогда уже останавливаемся и выносим решение обратно на координационный уровень.

## Итог

Лучший рабочий режим на текущем этапе такой:

1. здесь держим верхний уровень, проверки связности и интеграцию;
2. глубоко разрабатываем по одному продуктовому блоку в отдельных чатах;
3. после этого возвращаемся сюда для сборки общей системы без расползания в глубину.
